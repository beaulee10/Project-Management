### Project Statement
This app will be a sports analytics app that mainly tracks things like injuries, players sitting out, and different conditions like the weather as well and keeping data how those have affected players and teams in the past. For example, if a player is hurt on a team, whose stats become elevated during that time the player is out for injury or just sitting out. It will also take a look at the weather and keep stats that way. For example, if the weather is cold it may affect the game in different ways in football like things like passing as it is harder to throw the ball and harder to catch so the running back may have more opportunities. 
My project will be an app for different athletic programs. It is stat related where the app can see the stats for the players on whatever sport it is. The app will read in the file of the stats and see how players differentiate from each other such as when one player is injured, what stats of other players are higher. Or another example is may show the stats of players that seem to have good chemistry based on the numbers when they are in the same game together. This app will analyze players’ habits and stats to a whole another level.
### Technical debt 
Before we get into that there has been some recent research showing how serious technical debt has become. A study cited by IBM found that worldwide technical debt reached the equivalent of 61 billion days of repair time in 2025 (IBM, 2026). For Python specifically, the study found that it requires an average of 4.5 workdays to repair every 1,000 lines of code. (IBM, 2026). This means that my choice for python is great for your data analysis that we will use but as some point we have to prepare for maintenance cost.

While I have already mentioned Python we will also use react as they are both well known for having large communities. Therefore, when I run into problems, I may be able to find solutions online. This project is also open source so with these two large communities this will be another positive. Another positive about Python and React is that there have may pre-built parts so I don’t have to necessarily start from scratch. Using these common tools reduces one type of technical debt because I am not starting from scratch and inventing my own solutions for common basic problems
Instead of building two separate apps, I will use the same react code for both phones and computers (at least at the start). This will save money and time. IT also reduces technical debt because we only have to maintain one codebase. If I had two separate apps for computers and phones, then it would need fixed twice or some of them may have their own problems. Also new features would have to be built twice causing us to spend more money that we don’t need to spend.
Also, I will not try to sort every possible stat at the start for every sport. Instead we will start small for 3 main sports of our choosing. In doing this we will eventually get feedback and expand later but if we start with three that will cause us less technical debt. As the app grows and the users ask for more we will add more down the line, but we will start small and grow as time goes on. This approach reduces the amount of data I have to manage early on keeping the technical debt low.
When it comes to software updates python and react will release new versions over the next five years. With that being said, updates could break my existing code. For example, Python could change how a data library works, forcing me to rewrite parts of my analysis code. Each time I have to update, I lose time that I could spend expanding and building the app. The IBM report notes that organizations that ignore technical debt risk losing 18% to 29% of expected returns on their technology investments (IBM, 2026). That is a significant cost, but it is just part of running a real application.
If many people use my app at the same time, the single code base may run slowly and lag a little. Python has a limitation called the Global Interpreter Lock that makes parallel processing harder. React can also slow down if too much is happening at once. If this app becomes popular, I will eventually need to split it into a smaller services. That is a form of technical debt that will appear if the app succeeds though.
Sports data come from APIs which do charge money. For example, Sportmonks, a sports data provider, charges between 33 and 289 dollars per month depending on how many API calls you need (Sportmonks, 2026). Some start cheap and raise their prices later while others change their data formats with no warning which potentially breaks my code. If an important API goes away, I will have to spend months fixing it. To manage this risk, I would build an abstraction layer so that data sources look like the app. I will also try to keep raw copies of the data I get.

### Open-Source
With this app requiring many people to work on it and many people it will also be an open-source project. This is another way I will manage my technical debt. This will help create transparency as well. When the code is public on GitHub, other developers can spot mistakes I missed and notice a bug. Each bug caught before reaching users avoid technical debt. Another thing it does is attract contributors. While the team may be small to start, we will need help. If a coach wants baseball stats for example and I don’t have the time for it a volunteer who loves baseball may be able to add the feature. More contributors mean the maintenance is shared rather than everything falling on us. Another thing is that this project, being open source, will allow us to write cleaner code. Knowing that strangers might read my work makes me more careful. I may choose better comments of variable names and am able to break down different functions better. These habits can also help reduce technical debt as well. Open-source projects also allow for more collaboration with other people making things more reliable
### Database
Another thing that is very important to our project is our database. Our app revolves around data so our database must be elite. For our project we are going to use SQLite. For this there is not database server to install or maintain as the database is just a file. I would be able to avoid technical debt related to server management. Also, SQLite works everywhere as it it works on all platforms as it is built into IOS, Android, Windows, MacOS, and Linux. I do not also have to write different database code for different devices as well which helps us. Finally, one of the main reasons we are using this version is that it is reliable 

While this is an open-source project, we will still have a team for the app. There are a few different ways I would approach this. First, we would need a team of data analysts for different sports. We will also need people who are proficient in SQL as well as our database will be the main thing that revolves around this app. We would use python for our main code as well. We will also need marketers as well to get our app out there.  According to the SQLite documentation, the codebase has 100 percent test coverage and is one of the most reliable software components in existence (SQLite Consortium, 2025). The only downside to this is that SQLite is not built for a massive scale that our app could potentially produce. It cannot handle thousands of people writing code on the same database. Though this is fine for the early stages as I can change do a larger database later.
